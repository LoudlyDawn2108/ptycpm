## Key Principles

1. **Routes are thin** — route files only define `validateSearch`, `loader`, and `component`. They import from `features/` for all logic.

2. **Features are colocated** — each domain (employees, accounts, training...) has its own `api.ts` (query options + mutations), `hooks.ts`, and `components/`. This keeps related code together.

3. **`components/ui/`** is exclusively shadcn-generated — don't hand-write files here (except `data-table.tsx` and `combobox.tsx` which wrap shadcn primitives).

4. **`components/shared/`** is for cross-feature components used in 2+ features — don't put feature-specific components here.

5. **`lib/`** is for pure utility functions with no React dependencies.

6. **`stores/`** is exclusively for client-side state (auth, UI). Server data lives in TanStack Query cache.

## Naming Conventions (aligned with existing conventions.md)

| Type | Convention | Example |
|---|---|---|
| Route file | `kebab-case.tsx` | `bank-accounts.tsx` |
| Component file | `kebab-case.tsx` (shadcn style) or `PascalCase.tsx` | `employee-form.tsx` |
| Store file | `camelCase.ts` | `auth.ts` |
| Hook file | `use-kebab-case.ts` | `use-debounce.ts` |
| API/query file | `api.ts` (per feature) | `features/employees/api.ts` |
| Type file | `types.ts` (per feature) | `features/employees/types.ts` |
| Utility file | `kebab-case.ts` | `error-handler.ts` |
